# ADR-0001: Выбор стека

**Статус:** принято · **Дата:** 2026-06-30 · **Автор:** app-architect

## Контекст
Le Chef — мобильное кулинарное приложение с играми (колесо, квиз), распознаванием
продуктов по фото/голосу, рекомендациями, каталогом рецептов, ресторанами и
бронью шефа. Нужен стек, быстрый в разработке, с одной кодовой базой под iOS+Android
и удобной интеграцией внешних vision/STT-сервисов.

## Решение
- **Frontend:** React Native + **Expo** + TypeScript. Навигация — **React Navigation**,
  состояние — **Zustand** (лёгкий, без бойлерплейта).
- **Backend:** **FastAPI** (Python) + **PostgreSQL** + **SQLModel**.
- **Фото-распознавание:** внешняя vision-модель (API) → список ингредиентов.
- **Голос:** speech-to-text → текстовый поиск.
- **Фото-хранилище:** объектное (S3-совместимое); в БД только ссылка.

## Обоснование
- Expo: один код под обе платформы, быстрый цикл, OTA-обновления, доступ к камере
  для скана из коробки.
- FastAPI: асинхронность для внешних AI-вызовов, авто-OpenAPI (синхронизация с
  api-contract), Python — удобен для интеграций ML/vision.
- SQLModel: одна модель для ORM и валидации, меньше дублирования.
- Zustand вместо Redux: подсистемы (игры, бронь) развиваются независимыми срезами.

## Последствия
- Две экосистемы (TS + Python) — контракт между ними фиксируется в
  `docs/api-contract.md` (точка правды).
- Выбор провайдеров vision/STT/оплаты — отдельные ADR от backend-dev.

## Альтернативы (отклонены)
- Единый TS-монорепо (Node + Next.js + Expo): общие типы end-to-end, но хуже для
  ML/vision-интеграций и расходится с уже описанной зоной backend-dev.
