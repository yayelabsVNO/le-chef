# ADR-0003: Навигация (fun → utility) и точки расширения

**Статус:** принято · **Дата:** 2026-06-30 · **Автор:** app-architect

## Контекст
Продуктовая стратегия (см. docs/navigation.md): вход через игру (колесо/квиз),
плавный спуск к утилите (рецепт/корзина/бронь). Геймификация и бронь шефа —
отдельные подсистемы, которые будут расти независимо.

## Решение
1. **Контракт навигации** кодируется типами в `app/src/navigation/types.ts`
   (`RootTabParamList` + `RootStackParamList`). Это точка правды для экранов.
2. **Таб-бар фиксирован**: 5 иконок — home · категории · центральная кнопка-лого ·
   корзина · профиль.
3. **Инвариант «без тупиков»:** экраны `WheelResult` и `QuizResult` обязаны нести
   переход дальше (рецепт / скан / корзина / бронь). На уровне API это гарантирует
   поле `links` в ответах `/wheel/spin` и `/quiz/submit` (минимум одна ссылка).
4. **Точки расширения** изолированы в `app/src/features/`:
   - `games/` — колесо и квиз;
   - `booking/` — повод и бронь шефа.
   У каждой — свой README с инвариантами и список экранов/эндпоинтов.
5. **Петля персонализации:** результаты игр и сканов пишутся в профиль и питают
   экран `Recommendations`.

## Последствия
- Добавление экрана = правка `types.ts` + регистрация в `RootNavigator` (зона
  app-architect/согласование), затем вёрстка (frontend-dev).
- Новые игровые механики/типы брони добавляются внутри `features/*` без влияния на
  утилитарную часть.

## Открытые вопросы
- Провайдер оплаты брони (backend-dev, отдельный ADR).
- Офлайн-кэш рецептов (НФТ) — стратегия хранения на устройстве (отдельный ADR).
